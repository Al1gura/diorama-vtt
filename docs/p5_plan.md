# P5 CPR 战斗操作闭环开发计划

> 状态：2026-07-22 完成重新调研与任务拆分；P5 功能代码尚未开始。实施必须按 P5.1 → P5.5 顺序推进。后续任务可以提前调研，但前置依赖完成前不得修改后续阶段的功能代码。

## 一、阶段目标

让 GM（游戏主持人）在当前三维地点完成以下轻量闭环：

`备团设置 Token → 自动建立参战列表 → 一键或手动先攻 → 推进当前角色 → 按地形自动计算移动 → 取得远程攻击距离/DV/遮挡提示 → 退出战斗模式`

P5 是 CPR（《赛博朋克 RED》）规则辅助，不是完整角色卡、骰子平台或自动裁决引擎。

### 确认采用

- 当前地点中标为默认参战的 Token（标记）自动加入，不要求运行时逐个选中。
- CPR Token（《赛博朋克 RED》标记）一次定义十项主属性名称，避免后续反复迁移。
- 一键先攻使用 `REF（反应）+ 1D10`，同时允许 GM 手动填写和修改最终值。
- 不提供单 Token（标记）重投按钮。
- 同值时玩家优先是 Gvtt/本团简化规则，不冒充 CPR 原版；同阵营同值保持固定收集顺序，GM 可手改。
- 地形语义决定行走、困难地形、游泳、攀爬、跳跃/翻越；系统按规则成本选择路线。
- 远程攻击复用现有战斗线，只增加目标、距离和规则提示，不自动判定命中。
- 战斗面板只保留当前角色、下一位和必要提示；不做上一位、悔棋、动作合法性警告或战斗历史。
- 退出战斗模式只清临时队列、高亮、移动余额、目标模式和战斗线；不删除、不回滚 Token。

### 明确不做

- 完整角色卡、技能库、装备库、职业能力、赛博改造与属性修正管线。
- 通用骰子系统。P5 的批量 `1D10` 先攻是用户明确批准的窄例外。
- 自动闪避、命中、伤害、护甲削减、HP（生命值）、严重伤势、死亡或治疗。
- 单 Token 重投、上一位/上一轮、悔棋快照、动作记录和战斗专用保存恢复。
- DND（《龙与地下城》）规则实现；只保证射击提示接口将来可以接入其他规则。

## 二、现成资源核查与取舍

### 2.1 项目内直接复用

| 现有内容 | 当前能力 | P5 处理 |
|---|---|---|
| `CprTokenProperties` | 已保存 `move_stat` | 直接扩成十项 CPR 主属性，不另建第二套角色资源 |
| `TokenProperties` | 通用碰撞、移动和瞄准能力 | 增加阵营和默认参战标记，不混入 CPR 属性 |
| `PlacementController` | 新 Token 自动挂通用和 CPR 属性组件并设置 owner（归属） | 沿用挂载链；迁移旧 Token 时补缺失字段/组件 |
| `main.gd` 属性面板 | 已能编辑 MOVE（移速）并标记地图已修改 | 扩成可折叠十属性区和参战设置；后续可再拆控制器，不复制第二个面板 |
| `MovementService` | 导航、预算、路径截断、段标签和跟随 | 扩展地形加权择路与战斗剩余预算，不新建移动系统 |
| `TraversalProperties` | 可走、阻挡、困难、攀爬、跳跃、游泳 | 保留六类物理语义；增加“可翻越”的编辑说明或映射，不丢失跳跃/攀爬区别 |
| `CombatLinePreview/Query` | 三维射线、首个阻挡物和二元遮挡 | 增加射手/目标距离及规则提示，不重写射线 |
| 地点场景保存/迁移 | `ContentRoot` 打包、实例化、旧实体补组件 | 十属性和参战设置继续随地点保存，不新增平行 JSON 真值 |

### 2.2 外部方案

- Foundry CPR（Foundry 平台的《赛博朋克 RED》系统）仍在维护，采用十项属性与 `stats.ref` 一类稳定访问路径，可借鉴字段分层和迁移思路；其代码依赖 Foundry Actor（角色对象）并使用 GPL-3.0-only（仅 GPL 3.0）许可证，不直接复制进 Gvtt。
- Foundry Combat Tracker（战斗追踪器）的一键全体先攻和手动改值交互成熟；只复用交互模式，不引入联网、玩家权限、单体重投或完整角色系统。
- Magnus Laser（赛博朋克管理工具）为 MIT（宽松许可证）的 Tauri/网页项目，可参考离线 GM 工具边界，但不能替代 Godot 场景内 Token 属性与导航。
- 当前工程插件只有 `godot_ai` 和测试相关内容，没有可直接复用的 CPR 属性、加权地形、先攻或 DV 提示插件。

结论：采用“项目现有组件 + 社区成熟模式”，不安装新插件，不复制外部规则系统代码。

## 三、十项 CPR 主属性合同

`CprTokenProperties` 保存以下十个稳定字段；后续不得用另一套同义变量重复保存：

| CPR 名称 | 中文 | GDScript 字段 |
|---|---|---|
| INT | 智力 | `int_stat` |
| REF | 反应 | `ref_stat` |
| DEX | 敏捷 | `dex_stat` |
| TECH | 技术 | `tech_stat` |
| COOL | 酷 | `cool_stat` |
| WILL | 意志 | `will_stat` |
| LUCK | 运气 | `luck_stat` |
| MOVE | 移速 | `move_stat` |
| BODY | 体格 | `body_stat` |
| EMP | 共情 | `emp_stat` |

### 3.1 属性接口

字段是唯一持久真值；组件同时提供规则读取接口：

- `get_ruleset_id() -> StringName`：返回 `cpr`。
- `has_stat(stat_id: StringName) -> bool`：是否支持该属性。
- `get_stat(stat_id: StringName) -> int`：读取当前存储值。
- `set_stat(stat_id: StringName, value: int) -> bool`：校验后写入。
- `get_stat_ids() -> Array[StringName]`：按规则书顺序返回十项属性标识。
- `get_effective_stat(stat_id: StringName) -> int`：P5 暂时等于存储值，为后续护甲/伤势/赛博改造修正预留入口，但本阶段不实现修正管线。

不使用一个松散 `Dictionary（字典）`代替十个静态字段；这样场景保存、属性面板和旧数据迁移都有明确类型。

### 3.2 通用 Token 参战合同

`TokenProperties` 增加：

- `combat_side`：`PLAYER（玩家）`、`ENEMY（敌人）`、`NEUTRAL（中立）`。
- `auto_join_combat`：开始战斗时是否自动加入。

开始战斗只扫描当前地点 `ContentRoot` 下的 Token，不按幕收集，也不要求先选中。阵营和是否参战是两个概念：中立角色也可能参战，敌人也可能暂不登场。

## 四、P5 五个开发阶段

## P5.1 十属性与自动参战数据底座

**目标：** 备团时能为每个 Token 编辑十项 CPR 属性、阵营和默认参战状态；旧地图安全迁移并完整读回。

**实施内容：**

1. 扩展 `CprTokenProperties` 十项静态字段和规则读取接口。
2. 扩展 `TokenProperties` 的阵营与默认参战字段。
3. 属性面板增加可折叠的“CPR 属性”和“战斗设置”。
4. 旧 Token 自动补组件/默认值；迁移后标记地图待保存，不静默覆盖已存在的 MOVE。
5. 提供“扫描当前地点默认参战 Token”的只读查询服务，暂不显示战斗队列。

**复用：** 现有组件挂载、属性面板、地点场景保存和迁移链。

**不做：** 技能、HP、护甲、装备、属性修正、骰子和战斗面板。

**验收：** 十字段默认值/读写/非法标识、旧 MOVE 保留、旧场景迁移、地图保存重开、阵营和参战标记读回；MCP（运行态接口）验证面板回写到底。

**实施技能：** `godot`、`resource-pattern`、`godot-ui`、`save-load`、`godot-testing`。

## P5.2 地形语义与加权自动择路

**依赖：** P5.1 已完成且 `move_stat` 接口稳定。

**目标：** 点击目的地后，系统按实际移动预算选择行走、困难、水、攀爬和跳跃/翻越的最低成本合法路线。

**实施内容：**

1. 保留六类 `TraversalProperties` 语义，明确可翻越到跳跃/攀爬的映射。
2. 普通/困难/水域的成本进入导航择路阶段；攀爬/跳跃继续使用带成本的 `NavigationLink3D（3D 导航链接）`。
3. 路径结果返回分段方式、几何距离、规则成本和不可达原因。
4. 按 CPR 规则处理普通成本、双倍成本和立定跳距离限制。
5. 战斗外仍沿用整份移动预算；为 P5.5 预留“调用方传入剩余预算”的入口。

**必须验证的对照地图：** 穿水近路对比绕桥远路、困难地形捷径对比普通远路、单向/双向攀爬链接、助跑/立定跳边界、超预算截断、无合法路线。

**不做：** 擒拿拖行、伤势减速、载具移动、复杂角色动画。

**验收：** 自动路线确实按规则成本而非几何距离选择；路径段标签与费用一致；MCP 验证预览、点击和移动结果。

**实施技能：** `godot`、`ai-navigation`、`player-controller`、`math-essentials`、`godot-testing`。

## P5.3 自动/手动先攻与轻量队列

**依赖：** P5.1 完成；不依赖 P5.2，可提前完成调研但按顺序合入。

**目标：** 当前地点一键建立参战列表并完成先攻排序。

**实施内容：**

1. 自动扫描 `auto_join_combat=true` 的当前地点 Token。
2. “一键先攻”对每名参战者执行 `get_effective_stat(ref) + 1D10`。
3. 保留手动填写/修改最终先攻值；不提供单人重投。
4. 降序排序；跨阵营同值玩家优先，同阵营同值保持固定收集顺序。
5. 规则说明明确区分 CPR 原版同值重掷与 Gvtt 简化规则。
6. 队列只保存本次运行中的 Token 引用和临时先攻值，不做跨程序恢复。

**不做：** 通用骰子输入框、技能检定、上一位、悔棋、历史记录。

**验收：** 固定随机种子排序、手动覆盖、空列表、Token 删除、同值规则、连续多次进入/退出无残留；MCP 验证列表点击不穿透地图。

**实施技能：** `godot`、`godot-ui`、`state-machine`、`input-handling`、`godot-testing`。

## P5.4 通用远程攻击提示接口与 CPR DV

**依赖：** P5.1 完成；复用 P2.4 战斗线。

**目标：** GM 选择射手和目标后立即看到距离、遮挡、射程区间和 CPR DV（难度值）。

**通用提示结果至少包含：**

- 实际距离；
- 射程区间；
- 是否超距；
- 是否存在目标数值；
- 目标数值及类型；
- 正常/劣势/不可攻击等掷骰状态；
- 遮挡结果；
- 人类可读说明。

CPR 适配器返回武器类型、开火模式和距离对应的 DV；未来 DND 适配器可以返回对抗 AC（护甲等级）以及正常/劣势/不可攻击，不需要修改战斗面板合同。

**实施内容：** 目标选择、三维距离、CPR 单发 DV 表、现有二元遮挡组合、规则来源说明和 GM 可覆盖入口。全自动、霰弹和爆炸范围若会扩大本阶段，可拆成 P5.4b，但接口必须一次定稳。

**不做：** 自动闪避、命中、伤害、护甲或弹药结算。

**验收：** 所有射程边界、超距、遮挡、射手/目标排除、不同武器/模式；MCP 验证目标模式不误拖 Token、不误转相机。

**实施技能：** `godot`、`physics-system`、`input-handling`、`godot-ui`、`godot-testing`。

## P5.5 回合移动整合与退出闭环

**依赖：** P5.1-P5.4 全部完成。

**目标：** 把队列、剩余移动预算、自动路线和攻击提示合成 GM 可连续操作的一场战斗。

**实施内容：**

1. 显示轮次、当前角色和“下一位”。
2. 当前角色每回合获得一份 `MOVE × 2` 基础移动预算；分段移动累计实际规则成本。
3. “跑”作为明确操作增加第二份移动预算，但不建立通用动作合法性警告。
4. 非当前角色仍可由 GM 修正位置，但不扣当前角色预算，并明确显示为位置修正。
5. 当前角色切换时清理临时路线和目标模式，保留地图上的真实 Token 位置。
6. “退出战斗模式”清队列、高亮、预算、攻击目标和战斗线，不删除或回滚任何 Token。
7. 完成 P1-P4 全量回归、P5 专项回归、Windows 可见窗口与 GM 真人操作验收。

**不做：** 动作历史、回合回退、战斗专用存档、自动状态结算。

**验收主流程：** 自动收集 → 一键先攻 → 手改一项 → 推进角色 → 分两段移动并经过不同地形 → 查看远程 DV/遮挡 → 推进一轮 → 退出 → 再进入无旧状态残留。

**实施技能：** `godot`、`godot-ui`、`state-machine`、`input-handling`、`player-controller`、`godot-testing`。

## 五、每个 Codex 任务的统一工作方式

每个 P5.x 使用独立 Codex（编程助手）任务，第一轮固定执行：

1. 读取本计划、`docs/design.md`、`docs/roadmap.md`、`devlog/DEVLOG.md`和涉及的当前代码。
2. 完成四层调研：项目现状、Godot 4.7 源码、官方资料、英文社区/插件。
3. 公开调研回执、源码对照卡、可复用资源、风险、具体步骤和自动/运行验收。
4. 默认不修改功能代码；若前置阶段未完成，只提交研究结果和接口建议。
5. 用户在该任务中明确继续（包括单独回复 `1`）后，再复查前置依赖；依赖完整才实施、测试并更新日志。
6. 后续任务不得为了抢跑而改写前一任务仍在建设的公共接口；发现冲突先记录并回报。

## 六、四层调研与 Godot 源码依据

### 项目现状

- `CprTokenProperties` 只有 `move_stat`，没有第二套属性资源。
- `PlacementController::_attach_ruleset_token_properties()`已负责 CPR 组件挂载和 owner（归属）。
- `main.gd` 已有 MOVE 属性控件、回写和场景脏标记。
- `SceneSessionController` 在加载地点后调用实体迁移，再把地点内容作为唯一真值。
- `MovementService` 已有规则提供器、路径段标签、预算截断和链接成本；水/困难区域仍主要事后计费。
- `CombatLineQuery` 已有射线和二元遮挡，不应重写。

### CPR 规则

- 核心规则书 PDF p.90-97（书内 p.72-79）确认十项主属性及顺序：INT、REF、DEX、TECH、COOL、WILL、LUCK、MOVE、BODY、EMP。
- PDF p.144-145、186-187（书内 p.126-127、168-169）确认 `REF + 1D10` 先攻、`MOVE × 2`、同值重掷、分段移动和双倍通行成本。
- PDF p.190-192（书内 p.172-174）确认远程攻击按武器与距离查 DV。

### Godot 4.7 源码和官方资料

- 基线：`4.7-stable / 5b4e0cb0fd279832bbdd69fed5354d4e5ad26f88`。
- `scene/main/node.cpp::Node::set_owner()`决定节点是否归属并能进入场景保存子树。
- `scene/resources/packed_scene.cpp::PackedScene::pack()`/`SceneState::pack()`保存具有 `PROPERTY_USAGE_STORAGE（属性用于存储）`的字段；`SceneState::instantiate()`重建节点和属性。
- `core/io/resource.cpp::Resource::duplicate()`和 `resource_local_to_scene（资源随场景唯一）`说明嵌套可变 Resource 默认共享时需要复制；本项目继续使用现有 Node 属性组件，避免引入共享真值问题。
- `NavigationLink3D.travel_cost`和 `NavigationRegion3D.travel_cost`以距离乘数参与最短路；`NavigationPathQueryResult3D.path_types/path_rids`可识别链接段。

### 英文社区与许可证

- Foundry CPR：现行维护、十属性和迁移体系可参考；GPL-3.0-only 且运行时不兼容，不复制代码。
- Foundry Combat Tracker：采用一键全部先攻与手动改值；不采用单体重投、上一回合和完整角色/伤害管理。
- Godot 导航社区：重叠区域成本存在路线可靠性风险，必须用桥/水、普通/困难对照地图实测。
- 未发现与当前 Godot 4.7、GDScript、单机 GM 工具边界同时兼容且可直接安装的 CPR 插件。

## 七、完成判定

P5 只有同时满足以下条件才算完成：

- P5.1-P5.5 自动测试全部通过，且不破坏 P1-P4 回归。
- 当前对话具备 Godot MCP（运行态接口）时，相关界面、输入、场景树和运行状态均有 `game_eval（运行态求值）`及日志证据。
- Windows 可见窗口完成完整主流程。
- GM 真人确认属性编辑、自动移动、先攻与射击提示不会拖慢现场操作。
- `docs/design.md`、`docs/roadmap.md`、本计划和 `devlog/DEVLOG.md`保持一致。
