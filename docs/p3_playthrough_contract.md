# P3 最小带团会话合同

> 状态：2026-07-21 P3.2 已实现。项目现状、Godot 4.7-stable 源码、官方资料和英文社区/开源方案均已核查；当前界面采用“一份模组底本、多份独立桌记录”，点击“开始”后在启动窗口列出并进入各桌。最新专项证据为数据层 `39/39`、控制器 `39/39`、可见主界面 `66/66`。

## 一、目标与边界

P3.2 交付一条最小但真实的带团恢复链：

- 模组底本、一次带团会话和临时演出有不同所有者与保存位置。
- GM 从底本创建一次带团会话，在运行态移动 Token 或破坏墙体后保存。
- 关闭程序、重新打开模组并选择该会话后，恢复当前地点和上述变化。
- 从同一底本开始新会话时，不修改旧会话，也不修改底本。
- 切地点、离开运行态、关闭模组和退出程序时，先处理会话保存结果，再恢复底本或释放运行服务。

P3.2 不交付：

- 完整叙事进度、探索记忆、楼层、媒体播放进度或特效实例。
- 按对象字段计算的通用差异协议。
- 自动存档轮换、存档缩略图、会话搜索/排序或复杂会话管理界面。
- 把临时演出、播放/暂停、淡入淡出或一次性视觉节点写进会话。

## 二、三层状态

| 状态层 | 唯一真值 | P3.2 保存内容 | 不得保存 |
|---|---|---|---|
| 模组底本 | `ModuleManifest + _canonical/<location>.scn` | 地点清单和备团编辑结果 | 某次带团的 Token 位置、破墙、临时播放 |
| 带团会话 | `Playthrough + sessions/<session_id>/states/` | 当前地点和已访问地点的运行态场景快照 | 原始媒体、临时呈现节点、GM 工具节点 |
| 临时演出 | 后续玩家输出/媒体控制器 | P3.2 不跨进程保存 | 底本、会话长期状态 |

同一值只能有一个所有者：

- 编辑态场景脏标记属于 `SceneSessionController`，表示底本需要保存。
- 运行态会话脏标记属于 `PlaythroughController`，表示当前会话需要保存。
- `main.gd` 和按钮只显示状态，不保存第二份镜像真值。

## 三、采用每地点会话快照

P3.2 采用“每个已访问地点一份完整会话场景快照”，不采用对象差异：

1. 当前对象没有跨保存稳定的 `entity_id`，只有运行进程内 `ObjectID`，后者不能写入存档。
2. 官方对象字段方案要求每种对象提供场景来源、父路径和自己的保存字段；当前 Token、墙、光源、交互物体没有统一保存接口。
3. `ModuleIo.save_scene_tree()` 已通过 P1/P2 验证，能保存内容树、导出属性和变换，并排除 `gvtt_runtime_only` 节点。
4. 以 `location_id` 作为快照身份，不依赖显示名，也不需要在 P3.2 提前建立完整对象差异框架。

代价是单个地点的快照可能比差异文件大。P3 最小会话只保存访问过的地点；若 P8 实测显示空间或迁移成本不可接受，再在相同会话合同下引入对象稳定标识和版本化差异，不改变上层“保存/恢复会话”命令。

底本后续修改采用刻意简单的规则：一桌尚未访问的地点读取最新底本，因此后来补充的新剧情会自然进入该桌；已经访问过的地点继续读取该桌快照，保留已经发生的 Token、墙体和光源变化，不自动把新底本与旧桌进度合并。团员通常不会返回已经经历过的地点，这类合并属于小众事件，不值得为 P3.2 引入模组分支或对象差异协议。确有需要时由 GM 在该桌中手动补充，另行评估专门工具；模组备份只能保留底本恢复点，不能冒充会话合并。

## 四、磁盘结构

```text
user://modules/<本机目录名>/
├── manifest.json
├── _canonical/
│   └── <location_id 或旧文件名>.scn
└── sessions/
    └── <session_id>/
        ├── session.json
        ├── session.json.bak
        ├── session.json.tmp
        └── states/
            ├── <location_id>.scn
            ├── <location_id>.bak.scn
            └── <location_id>.tmp.scn
```

快照临时/备份文件仍以 `.scn` 结尾，保证 `ResourceSaver` 和 `ResourceLoader` 能按 `PackedScene` 识别。`session.json` 只保存相对路径，不保存本机绝对路径或 `user://` 展开结果。

## 五、session.json

第一版结构：

```json
{
  "format": "gvtt_playthrough",
  "schema_version": 1,
  "session_id": "22222222222222222222222222222222",
  "module_id": "0123456789abcdef0123456789abcdef",
  "session_name": "默认带团",
  "current_location_id": "11111111111111111111111111111111",
  "location_states": [
    {
      "location_id": "11111111111111111111111111111111",
      "state_relpath": "states/11111111111111111111111111111111.scn"
    }
  ],
  "notes": ""
}
```

约束：

- `format` 必须精确为 `gvtt_playthrough`，`schema_version` 必须是支持的正整数。
- `session_id`、`module_id` 和所有 `location_id` 必须是 32 位小写十六进制稳定标识。
- 目录名必须与 `session_id` 一致；`module_id` 必须与当前模组清单一致。
- `current_location_id` 必须引用清单中的现有地点；空模组才允许为空。
- `location_states` 中地点不可重复，只能引用当前清单地点。
- `state_relpath` 规范化后必须仍位于当前会话的 `states/` 下，并以 `.scn` 结尾；禁止绝对路径、`..` 和目录逃逸。
- 快照缺失不是 JSON 结构损坏，但恢复该地点必须返回明确的 `ERR_FILE_NOT_FOUND`，不得静默退回底本并冒充会话已恢复。
- 未来版本返回 `ERR_UNAVAILABLE`，不覆盖、不降级。

## 六、内存映射

`Playthrough` 继续作为内存中的强类型数据容器，磁盘格式改为校验后的 JSON：

- `const SCHEMA_VERSION: int = 1`
- `schema_version: int`
- `session_id: String`
- `module_id: String`
- `session_name: String`
- `current_location_id: String`
- `location_states: Dictionary`，运行时为 `{location_id: state_relpath}`
- `notes: String`

旧字段只作为迁移输入：

- `module_path` 不再持久化；由当前模组目录和 `module_id` 验证替代。
- `current_location` 显示名迁移为 `current_location_id`。
- `visited` 的“显示名 → 绝对/资源路径”迁移为 `location_id → 会话内相对路径`。
- `historical_notes` 迁移为 `notes`。

## 七、职责边界

### ModuleGate

- 持有当前 `ModuleManifest` 和当前 `Playthrough` 唯一引用。
- 列出、创建、打开、关闭会话，并返回结构化结果。
- 不直接打包场景树，不操作 GM 界面节点。

### PlaythroughController

- 普通节点，由 `Main` 创建并注入内容根、`ModuleGate` 命令和 `ModuleIo` 接口。
- 持有运行态会话脏标记。
- 协调“保存当前地点快照、恢复地点、切地点、开始新记录、离开会话”。
- 在任何清理/Token 底本恢复之前发起快照保存。

### SceneSessionController

- 继续只负责编辑底本：新建地点、加载 `_canonical`、保存 `_canonical` 和编辑脏标记。
- 不保存会话快照，不持有当前 `Playthrough`。

### ModuleIo

- 编解码、校验和可恢复写入 `session.json`。
- 打包、验证、提交和恢复会话 `.scn` 快照。
- 读取快照使用 `ResourceLoader.CacheMode.CACHE_MODE_IGNORE`，避免覆盖同路径后取到旧缓存。

### Main

- 作为组合根注入依赖并把用户命令路由到控制器。
- 编辑态保存按钮显示“保存场景”，只写模组原始场景。
- 编辑态提供“测试”和“开始”两个运行入口；“测试”不创建或读取会话，“开始”自动决定创建或读取当前记录。
- 运行态变化延迟自动保存到当前会话；不向 GM 暴露单独的“保存带团”动作。
- 顶栏不显示“开始带团 / 继续带团 / 选择带团”；多份记录的选择仍是记录管理行为，不改变“开始”按钮文案。
- 不自己拼会话路径、序列化字段或维护第二份脏标记。

## 八、工作流与调用顺序

### 打开模组

1. P3.1 完成清单加载/迁移并提交 `ModuleGate` 当前模组。
2. 保持编辑态，加载清单起始地点底本。
3. 扫描 `sessions/<session_id>/session.json`；点击编辑态顶栏“开始”后，在“开始带团”窗口展示全部记录，合法项显示“进入”，损坏项显示错误并禁用进入，不自动覆盖。
4. 打开模组不立即替换编辑内容；编辑态固定显示“测试”和“开始”。

### 测试

1. 把当前内存内容根打包为不落盘的 `PackedScene（打包场景）`，并先实例化验证。
2. 不要求保存当前编辑改动，不创建或读取 `session.json`，直接进入运行态。
3. Token、墙体和光源变化不标记模组场景，也不启动会话自动保存。
4. 返回编辑态时清理运行服务，使用内存快照完整恢复测试前内容和原编辑脏状态。
5. 选择模组时先恢复测试前内容；关闭程序时可直接释放测试快照，不写入任何持久层。

### 开始（内部无现有记录）

1. 生成新 `session_id`，建立内存 `Playthrough`，不复用旧 ID。
2. 使用安全写入创建初始 `session.json`；失败时不进入运行态。
3. 加载 `start_location_id` 的底本，不创建地点快照。
4. 提交当前会话并进入运行态；之后的变化只属于新会话。

### 开始（内部已有记录）

1. 在局部变量中读取、恢复、迁移和验证 `session.json`。
2. 验证 `module_id`、当前地点和路径，不先替换当前会话。
3. 当前地点已有快照时加载并验证快照；没有快照表示从未保存该地点，加载底本。
4. 场景加载成功后一次性提交当前会话与内容树，再进入运行态。
5. 任一步失败都保持原模组、原模式和原内容树。

### 自动保存当前会话

1. 取消未完成的拖动，但不恢复 Token 编辑态快照。
2. 暂停会继续修改内容的运行服务。
3. 把当前内容根写到 `<location_id>.tmp.scn`。
4. 用忽略缓存模式读回、实例化并验证临时快照，然后释放验证树。
5. 备份并提交快照；失败时立即恢复原快照。
6. 在 `Playthrough` 副本中更新 `current_location_id/location_states`，安全写入 `session.json`。
7. JSON 成功后才提交内存会话并清除会话脏标记；失败时保留旧内存真值，固定路径的新快照作为可恢复数据保留。
8. Token、墙或光源变化后由一次性定时器延迟保存；Token 仍沿路径移动时短暂重试，不能为了自动保存截停移动。

### 运行态切地点

1. 若当前会话脏，先执行“保存当前会话”；失败则取消切换。
2. 停止移动、瞄准、LOS 临时引用和未完成手势。
3. 目标地点有会话快照则加载快照，否则加载底本。
4. 加载成功后替换内容树、更新 `current_location_id` 并保存 `session.json`。
5. 重建目标地点的运行服务；失败时报告，不把底本写成会话恢复结果。

### 回到编辑态

1. 若会话脏，先保存当前会话；失败则保持运行态并提示重试/放弃/取消。
2. 停止运行服务和临时演出。
3. 重新加载同一 `location_id` 的底本，而不是只恢复 Token 位置；这样墙体、光源等会话变化也不会留在编辑底本视图。
4. 切到编辑态，编辑脏标记从底本加载结果开始为 false。

### 关闭模组或退出程序

1. 取消未完成输入和异步请求。
2. 当前会话脏时先保存；失败必须返回给应用退出协调层，不得静默退出。
3. 关闭玩家输出和临时演出，释放移动/LOS/战斗临时服务。
4. 清空内容树，再清除 `ModuleGate` 当前会话和模组引用。

## 九、可恢复快照写入

场景快照与 JSON 不是一个文件，Godot 不提供跨文件事务。P3.2 固定顺序：

1. 快照 `.tmp.scn` 写入、读回、实例化验证。
2. 旧正式快照复制为 `.bak.scn`。
3. `.tmp.scn` 提交为正式 `.scn`；失败时从备份恢复。
4. 正式快照成功后再安全写 `session.json`。

这个顺序保证 JSON 不会先指向尚未提交的快照。首次保存若在第 3、4 步之间中断，会留下尚未登记的孤立快照；下次会话扫描可报告并清理，不能自动把它加入会话。Windows 重命名仍有短暂中间窗口，因此只承诺“正式/备份/临时至少一份可验证并可恢复”，不宣称断电级原子性。

## 十、错误结果与回退

沿用 P3.1 结构化结果：

```text
{
  error: int,
  value: Variant,
  recovered_from_backup: bool,
  migrated: bool,
  message: String
}
```

额外规则：

- 会话 JSON 完全损坏：尝试备份；仍失败则拒绝打开，不创建同 ID 空会话。
- 当前地点快照损坏：尝试 `.bak.scn`；仍失败则拒绝恢复该地点，不静默加载底本。
- `module_id` 不匹配：返回 `ERR_INVALID_DATA`，不得把其他模组会话接到当前底本。
- 保存失败：不清脏、不切地点、不切模式、不关闭模组。
- “放弃本次未保存变化”只能由明确用户命令触发；它重新加载上次会话快照或底本，不覆盖磁盘。

## 十一、测试矩阵

### P3.2 专项自动测试

- 新会话有合法稳定 ID、版本、模组 ID 和起始地点 ID。
- 新会话创建前后，清单与底本场景字节不变。
- 运行态移动 Token、破坏墙并保存后，快照包含变化且不包含 `gvtt_runtime_only` 节点。
- 关闭整棵测试应用树并重新实例化后，继续会话恢复当前地点、Token 位置和墙体状态。
- 未访问地点从底本加载；保存后进入 `location_states`；再次切回读取会话快照。
- 回编辑态重新加载底本，Token、墙体和光源不保留会话变化。
- 从底本新开会话生成新 ID，旧会话文件和状态不变。
- 重命名地点显示名不影响会话引用。
- 会话 JSON 损坏时恢复备份；未来版本和模组 ID 不匹配时拒绝且字节不变。
- 当前地点快照缺失/损坏时明确失败，不静默回退底本。
- 快照提交失败或 JSON 提交失败时不清脏、不切地点，旧会话仍可恢复。
- 编辑态保存只写底本；运行态保存只写会话。
- 测试运行退出后恢复 Token、墙体和光源，且不创建会话、不改变底本字节和原编辑脏状态。
- 切地点、回编辑态、关闭模组和退出程序均在 Token 恢复/内容清理前完成会话保存。

### 可见窗口验收

1. 编辑态显示“测试”和“开始”；测试中移动 Token、破坏墙、关闭光源，等待超过自动保存延迟后返回编辑态，确认变化消失且没有会话文件。
2. 点击顶栏“开始”，在“开始带团”窗口用“＋ 新增一桌”分别创建“周五团”和“周六团”；再次点击“开始”时两桌名称与“进入”按钮同时可见，选择前保持编辑态。
3. 记录运行中移动一个 Token、破坏一面墙，不点击保存，等待自动保存。
4. 切到第二地点，再切回，确认 Token 和破墙状态恢复。
5. 回编辑态，确认模组原始 Token 和墙体仍是备团状态，按钮仍显示“开始”。
6. 关闭程序、重新打开模组，点击“开始”仍列出周五团和周六团；点击对应“进入”分别恢复各桌。新增桌继续位于启动窗口，不出现在文件菜单。

## 十二、四层依据与取舍

- 项目：实现前缺口是 `Playthrough`/`ModuleIo` 骨架未接线、会话索引、保存路由和生命周期顺序；现已由 `PlaythroughController`、版本化 `session.json`、每地点快照和“开始”启动窗口接通并通过专项回归。
- Godot 4.7 源码：`PackedScene::pack()` → `SceneState::pack()`/`_parse_node()` 保存 owned（有归属）节点；`SceneState::instantiate()` 重建；`ResourceLoader` 忽略缓存模式用于覆盖后重读；`Node::set_owner()` 强制祖先归属。
- 官方：`Saving games` 的对象字段方案要求逐对象识别与序列化；`PackedScene` 明确支持保存节点及其 owned 子树到 `user://`。P3.2 因缺少稳定对象 ID 采用地点快照，未来可替换内部格式。
- 社区/开源：Tabletop Club 证明素材/预制游戏与可恢复桌面存档应分离，但未取得可靠内部源码，不宣称照搬实现；SaveState Lite 只借鉴版本、临时文件和备份，不安装；社区运行时 PackedScene 快照只作为可行性旁证。
