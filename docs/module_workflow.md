# Gvtt 模组工作流

## 启动与进入模组

每次启动 Gvtt 都先进入独立的 `module_home.tscn` 模组选择场景。这个轻量入口不创建 3D 编辑器、投屏视图或模型缓存。GM 必须新建、打开、导入或恢复一个模组，确认当前工作上下文后才切换到 `main.tscn` 场景编辑器。

进入模组后，模组选择场景被卸载，编辑顶栏不提供“模组”按钮。当前模组名称只作为不可点击状态显示；场景编辑工具只处理当前模组内的场景、视角和运行模式。模型缓存也只在成功进入编辑器后开始预热。

## 文件菜单

编辑器顶栏的“文件”菜单只包含两个模组文件命令：

- “保存模组”：保存当前 `.scn PackedScene（打包场景）`，再校验并安全写入 `manifest.json`。两步都成功后才清除未保存状态。
- “选择模组”：当前场景有未保存修改时先保存；保存失败则留在当前模组。成功后清空当前模组真值并回到独立选择页。退出整个程序仍使用窗口关闭按钮。

编辑器左栏只保留备团内容，不常驻“带团记录”。点击顶栏“开始”后打开独立的“开始带团”窗口，直接显示当前模组下的“周五团”“周六团”等记录；每行“进入”只恢复该桌进度，“＋ 新增一桌”输入桌名并创建独立记录。它们不是文件命令，不复制或改名模组，也不读取其他桌的变化。该窗口是进入记录运行的明确入口，不是隐藏小菜单。

模组选择页不是编辑器内可以临时打开再返回的工具面板。返回选择页表示当前模组上下文已经结束。

旧版本已经产生的 `_backups/` 数据保持原样，不自动删除；当前界面不再创建手动模组历史。`manifest.json.tmp/.bak` 继续只用于保存失败时自动恢复，不作为用户管理的版本。

## 开发与正式启动

Godot 编辑器可见运行、无窗口自动回归和正式导出 EXE 使用同一启动合同：都从模组选择页开始，不自动进入“开发测试模组”。测试需要模组时，由测试夹具显式创建或打开，避免开发便利逻辑改变正式产品行为。

## Godot MCP 开发边界

Godot MCP（模型上下文协议）按任务连接，不作为所有开发工作的常驻前置。它的价值是读取真实编辑器、场景树和运行态，代价是额外工具调用、日志上下文、等待时间、端口和编辑器状态维护。

| 工作类型 | 默认工具 | 完成门槛 |
|---|---|---|
| 文档、JSON、清单、源码搜索、静态检查 | 文件工具和命令行 | 文件依据与格式检查 |
| 模组/场景存档清理 | 文件系统工具；操作前关闭 Godot | 目标目录与资产保留复核 |
| 纯数据合同和迁移 | 自动测试，必要时无窗口 Godot | 可重复断言通过 |
| UI、场景树、拖放、输入、Gizmo、3D 视口 | Godot MCP + 自动回归 | `game_eval`/场景树实证 + Windows 可见验证 |
| 投屏、图片、视频、音频和窗口生命周期 | Godot MCP + Windows 专项 | 运行态状态、清理链和玩家窗口隔离通过 |
| 手感、布局、声音和桌边完整流程 | 可见 Godot + GM 真人操作 | GM 明确确认；自动测试不能替代 |

推荐流程：文件修改 -> 自动回归 -> 按需连接 MCP -> `helper_live`/日志通道确认 -> `game_eval` 与场景树实证 -> Windows 可见窗口 -> GM 真人验收。

当前对话没有暴露 `project_run`、`game_eval`、`helper_live`、`logs_read` 或 `script_manage` 时，必须明确报告 MCP 未连接。可以继续完成文档、文件实现和自动回归，但涉及界面或运行态的工作只能标为“待恢复 MCP 后验收”，不能把直接启动 Godot 测试进程说成操作了 Godot 编辑器。

## 自动回归

P1 无窗口回归会直接实例化 3D 编辑器，以覆盖既有编辑功能；它同时检查项目主入口确实指向独立模组选择场景、“文件”菜单两个模组命令、编辑左栏不混入带团记录、没有当前模组时编辑和模型预热均保持锁定。P3.1 专项回归单独覆盖清单、稳定标识、迁移、旧恢复点兼容、路径防护和失败回滚。P3.2 可见回归覆盖点击“开始”后同一窗口列出“周五团/周六团”、两份记录互相隔离，且底本不被任一桌改变。
