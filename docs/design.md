## 一、产品五维度

所有功能从这五个维度推导。画面品质不是第六维度——它是所有维度的底线约束。

| # | 维度 | 一句话 | 受众 |
|---|------|--------|------|
| ① | **场景呈现** | 玩家能看到立体的、空间上可信的场景 | 玩家 |
| ② | **场景因果** | 场景变化是真实的——因果关系，不是假装 | 玩家 |
| ③ | **模组管理** | GM 管理的是一个完整冒险，不是一张张孤立的地图 | GM |
| ④ | **交互逻辑** | 跑团中软件不打断游戏——GM 一气呵成 | GM |
| ⑤ | **准备成本** | GM 备团的投入可控——不被工具拖累 | GM |

### 五维度独立性验证

- ①呈现 vs ②因果：静态呈现 vs 动态变化真实性。2D VTT（虚拟桌游平台）能做①但永远做不到②
- ②因果 vs ④交互：④是 GM（游戏主持人）能执行操作，②是操作后世界真的变了
- ③模组 vs ⑤成本：管理能力 vs 时间投入，不同维度
- ①呈现 vs ⑤成本：GM 想给玩家看的 vs 为这个要花多少时间——核心张力
- ④交互 vs ⑤成本：直觉操作 vs 操作耗时，不是同一件事

## 二、产品边界

**是什么：** 用 Godot（开源游戏引擎）引擎做的 GM（游戏主持人）桌面工具——开源 3D 俯视交互地图引擎。Windows 单目录便携应用，GM 一人操作。

**第一场景：** 线下跑团，GM 电脑操作，投屏给全桌看。**线上功能初期不做，预留接口。**

**明确不做：** 联网同步、自动掷骰、完整自动规则引擎、模组市场、玩家端、账号。

模组首页的位置、开发测试自动进入规则和正式 EXE 启动规则见 [模组工作流](module_workflow.md)。

### 2.1 CPR 规则资料查阅门禁

首个模组使用 CPR（《赛博朋克 RED》）规则。Gvtt 不做完整自动规则引擎，但只要功能涉及 MOVE（移速）、移动动作、格距、先攻、伤害、护甲、技能、状态或装备效果，就必须先查本地 CPR 资料包，不能凭印象或通用游戏经验实现。

查阅顺序：

1. 先读 `docs/cpr_reading/agent_index/agent_readme.md`。
2. 先查 `agent_keyword_index.json`，再读取对应的 `agent_chunks.jsonl` 正文块。
3. 结论必须记录资料名、PDF 页码；如正文块标记 `needs_pdf_check=true`，或涉及复杂表格、图标、分栏，必须回看 `source_pdf` 指向的原 PDF 页面。
4. `cyberpunk_red_core.md` 与 `night_city_catalog.md` 用于上下文复核；不能只根据搜索命中的单句设计规则。
5. 用户明确要求偏离 CPR 时，必须把它记录为 Gvtt/模组自定义规则，不能写成 CPR 原规则。

核心规则检索真值是 `docs/cpr_reading/agent_index/agent_chunks.jsonl`。以后任何涉及游戏规则的实现，无论当前模组还是新模组，都必须先查所选规则资料的结构化来源；没有来源时只能做通用几何能力，不能凭经验补规则。

“不做完整规则引擎”只表示不自动包办掷骰、伤害结算和整套战斗流程，不表示可以忽略选定功能所依赖的 CPR 规则。经用户确认的局部规则辅助可以实现，但必须有上述来源依据。

**许可证：** MIT

### 2.2 Windows 发布形态

- 对用户只提供一个 ZIP 下载物；解压后直接运行，不需要安装程序。
- 内置脚本、场景和基础资源保持在 Godot PCK（资源包）中并随 EXE 发布。
- 原生扩展 DLL（动态链接库）放在 EXE 同目录；用户模组和媒体继续放在用户内容目录。
- “一个下载物”不等于“运行时物理单文件”；严格单 EXE 不再是产品要求。

## 三、编辑态 ↔ 运行态（+ 投屏窗口）

一体化的 Windows 单目录便携应用，制作、测试和正式带团可随时往返，不设置“制作完成”门槛。

- **编辑态**（备团）：地面材质绘制、物件拖放 + 属性标记、场景保存/加载
- **测试**（不记录运行）：临时使用运行态全部操作；退出后恢复测试前的内存场景，不创建带团记录、不写模组场景
- **开始**（记录运行）：打开“开始带团”窗口，选择已有记录或新增一桌后进入运行；Token（标记）、墙体和光源等变化延迟自动保存
- **投屏窗口**（玩家看的画面）：运行态场景的同步呈现窗口，纯 3D 场景，无 GM 控件

编辑态仍只保存模组原始场景；“测试”不写文件；“开始”只写选中的带团记录。两种运行方式复用同一套运行操作和投屏，不增加第三个 `AppMode（应用模式）`。编辑器左栏只放备团内容；点击顶栏“开始”后打开明确的“开始带团”窗口，集中列出“周五团、周六团”等独立记录并提供“新增一桌”。它不是隐藏小菜单，也不把记录误作模组版本。

### 3.1 为什么投屏窗口是「第三块」但不是第三个 AppMode

软件里实际有三块呈现：编辑态、运行态、投屏窗口。但从架构状态机角度，**编辑态/运行态是「GM 操作权限」的两档**，投屏窗口是运行态的**输出层**——不是并列的第三个权限档。

打比方：PPT 双屏。GM 笔记本屏幕 = 运行态（有缩略图、控件、文案），电视/投影 = 投屏窗口（纯画面）。两边看的是同一份 PPT，不是两份独立文件。GM 在运行态拖 Token、砸墙、关灯，投屏窗口实时同步看到。

所以投屏窗口旁路于 ModeGate（编辑↔运行两态闸），可单独打开/关闭，两个态都能投屏。当前实现使用一个独立原生 `Window（窗口）`，直接挂一颗只读 `Camera3D（三维相机）`，并显式共享主场景的 `World3D（三维世界）`；不额外套 `SubViewport（子视口）`。迷雾遮罩内部另用子视口绘制二维遮罩，二者不是同一层。详见 `docs/architecture.md`。

### 3.2 边界澄清：投屏窗口 ≠ 玩家端

第 38 行「不做：玩家端」指联网的独立玩家客户端（需要账号、配对、同步协议），与本设计无关。投屏窗口是 GM 本机上的第二个本地窗口，不联网、不需账号，本质是「输出到第二屏/共享单窗给会议软件」。这不违反现有边界；基础投屏已提前落地，P3 建立输出、数据和生命周期合同，正式媒体演出归 P4，CPR 战斗操作闭环归 P5；三维气氛与特效后移为第一版后候选。

### 架构约定：按用途拆分碰撞表征（2026-07-17 修订）

点击、移动、战斗和视线分别需要不同精度与生命周期，因此各管各的事，通过 Godot 的物理层（collision layer/mask）或独立导航世界隔开。不许让一个盒扛起所有事，也不许把移动、战斗或迷雾判定塞进点击代理。

用户选中物体时看到的橙色 Gizmo（操纵框）不是 `PickProxy` 本身。橙色框会把各模型的局部 AABB（轴对齐包围盒）折回物体根坐标后合并，再随物体根一起旋转；移动服务现在采用等价算法计算阻挡物局部边界。因此两边看到的基础轮廓应一致，但目前各自计算、没有直接共用同一个缓存：Gizmo 管编辑显示，移动服务的缓存管运行态导航和碰撞。

墙、汽车等非地形阻挡物使用随根旋转的 `BoxShape3D`（盒形碰撞体）；地形、坡面和台阶使用三角网格，保留真实高度。Token（标记）不能只看模型外观猜胖瘦，而是读取每个 Token 保存的碰撞半径和高度，并按体型复用对应导航图。

| 层/用途 | 组件或服务 | 几何表征 | 职责 | 阶段 |
|----|------|-----------|------|------|
| **拾取层** | PickProxy（点击代理，Area3D + BoxShape3D） | 宽容点击盒，可与橙色 Gizmo 不完全相同 | GM 点击选中物件 | P1 已落地 |
| **移动层** | MovementService（移动服务） | 非地形为缓存局部盒并保留旋转/缩放；地形为三角面；按 Token 体型缓存导航图 | 绕障、贴地、体积防穿模 | P2.2 已落地 |
| **共享遮挡几何源** | OcclusionGeometry（遮挡几何工具） | 从真实模型计算物件根局部边界 | 只提供源几何，供战斗层与迷雾层各自转换；不持有物理体、不执行查询 | P2.4 建立，P2.5 复用 |
| **战斗层** | CombatBody（战斗碰撞体） | 跟物件旋转的盒/多边形 | 挡枪线，不计算命中与伤害 | P2.4 |
| **迷雾层** | LOSOccluder（视线遮挡体） | 俯视投影到地面的遮挡轮廓 | P2.5 建立基础视线遮挡，P7 扩展规则与表现 | P2.5 基础 / P7 扩展 |

**引擎依据：** Godot 射线查询天生支持 `collide_with_areas`（查 Area3D）/`collide_with_bodies`（查物理刚体）两套开关 + `collision_mask` 按层筛（离线文档 gdd_0255 第144-221行）。点击选中查拾取层、挡枪线查战斗层、各查各的，互不干扰，不用自己造轮子。

**当前落地：** P1 已做拾取层；P2.2 已做独立移动层。PickProxy 职责边界写死，只管点击选中代理；移动、战斗和迷雾各有自己的服务或组件。物件根下已挂 EntityProperties + PickProxy，后续增加 CombatBody/LOSOccluder 不复用点击盒。

**P2.4/P2.5 共用边界（2026-07-18）：** 两阶段共用“遮挡源”概念、物件生命周期和 `OcclusionGeometry` 计算出的物件根局部边界，但不共用运行时组件、查询接口或开关。P2.4 把局部边界转换为独立 `StaticBody3D + CollisionShape3D`，用一次 3D 射线回答“这条枪线先撞到谁”；P2.5 把同一局部边界投影为地面遮挡线段，再计算整片二维可见区域。`WallProperties.blocks_shot` 与 `blocks_los` 始终独立，以支持玻璃、烟雾和矮掩体等不同语义。

**P2.5 避免重复工作约束：** P2.5 必须直接复用 P2.4 建立的 `OcclusionGeometry`，不得再次实现模型子树扫描、局部边界合并或旋转/缩放换算；也不得读取 `CombatBody` 的运行时碰撞形状作为 LOS 真值。P2.5 只新增“局部边界转地面轮廓、可见区域计算、迷雾显示”三段职责。

**移动层精度与缓存（2026-07-17）：** 源几何每次重建只解析一次，水平导航单元为 0.1 米、垂直单元为 0.25 米；导航图按 Token 的 `(collision_radius, collision_height)` 体型键按需烘焙并缓存。默认 `0.45 × 1.8` 米 Token 使用量化后的 `0.5 × 2.0` 米档；真实胶囊扫掠若在墙角拒绝基础路线，才查询半径增加 0.2 米的安全档。退出运行态、切场景或重建时显式释放缓存地图。

**点击错位修复（2026-07-14）：** PickProxy 旧版在 `_ready` 里立刻算盒，但此刻模型 `instance.scale` 刚设、还没传播到子节点 `global_transform`，算出错误盒中心（实测模型几何中心 (0,0,0)、盒中心却算成 (-2.45,5,-0.03)，屏幕偏左 94px → 只有右上方能选中）。改由 main.gd `_place_model` 放完物件后 `force_update_transform`（gdd_0673 第358-362行）强制同步 transform，再调 `fit_from_target_synced` 扫真实世界 AABB 贴盒。game_eval 双轮验证：盒中心对齐模型中心（screen_offset=0）、多点射线全命中。

### P2.4 可视验证：可锁定的自由方向战斗线

旧的“每个 Token 沿本地 `+Z` 发出固定 10 米红/绿探针”和“终点吸附实体/地面”的方案均已撤回。当前是 P2.4 临时验证交互：只选一个 Token 作为射手，从视觉模型顶部向下 `1/9` 的近似眼睛高度出发，鼠标只决定屏幕方向，固定预览 20 米世界线；它不把 20 米解释为武器射程或最终可射击结论。

**运行态边界（2026-07-18）：** 战斗线是跑团操作，不是备团编辑工具。编辑态选中 Token 只显示编辑 Gizmo 和属性面板，不进入战斗瞄准、不显示半圆/射击线、不锁线，也不拦截普通拾取；切到运行态后选中 Token 才启动 `CombatLinePreview`。

#### 碰撞层与组件结构

- 第 20 物理层只给 `PickProxy`（拾取代理）的 `Area3D` 使用；第 21 物理层只给 `CombatBody`（战斗碰撞体）的 `StaticBody3D` 使用。两者不能互查，也不得用 PickProxy 的 AABB（轴对齐包围盒）作为挡枪依据。
- 每次放置或读回模型时，墙体、交互物体、地形、装饰都会生成一个候选 `CombatBody`；Token 和光源排除。候选组件按真实视觉高度自动分类：严格高于 `1.0` 米才进入第 21 战斗层，低于或等于 `1.0` 米不挡枪。
- `CombatBody` 分开保存真实视觉边界、`blocks_shot` 和 `provides_full_cover`。旋转和缩放跟随物件根；运行物理体保留真实高度，不再把低车补高到 Token 身高；运行物理体不接收输入，也不写入场景存档。
- `CombatLineQuery` 是无界面查询接口；`CombatLinePreview` 是 `Main` 下唯一一个 GM 可视预览器。预览器不是 Token 子节点，因此选中两个 Token 也只会有一条线，旋转 Token 时会隐藏且不进入 Gizmo 包围范围。

#### 射线查询合同

- 选中的 Token 是射手。预览起点取视觉模型顶部向下 `1/9` 的近似眼睛位置；没有视觉边界时才回退到 `TokenProperties.collision_height` 的 `8/9` 高度。
- 鼠标只提供相对眼睛屏幕点的方向，终点固定为该水平世界方向 20 米处；不查询鼠标下实体、不投影到地面、不吸附地图边缘。屏幕上以眼睛投影点为圆心显示 96 像素上半圆，作为临时方向参照。
- 在视口按一次左键锁定当前世界起点和终点；锁定后移动鼠标、右键转相机或旋转模型都不能改线。再次按视口左键或第一次 `Esc` 解锁。
- 2026-07-18 输入修正：选中 Token 后进入持续瞄准模式，视口左键无论落在空白、汽车还是其他实体上，都只锁定/解锁战斗线，不再执行拾取。锁定时第一次 `Esc` 解锁，活动瞄准时再按一次 `Esc` 退出模式；退出后左键才恢复选择实体。
- 查询只检测第 21 战斗物理层、只检测 PhysicsBody3D（物理体），并排除射手与明确目标自己的 CombatBody RID（物理资源编号）。此外，已被自动分类为掩体的 `CombatBody` 会额外用“随物件旋转的 XZ 地面占用框”做二元挡线判断，因此测试线即使高于汽车真实高度，只要水平投影穿过汽车粗框，也会被判定为被掩体挡住。普通 Token 的 PickProxy 不会挡枪，目标本身也不会被误报为目标前遮挡。
- 青色整线表示该 20 米测试线内没有登记的 CombatBody；橙色到首个遮挡点、灰色到测试终点表示途中遇到了登记的战斗遮挡。颜色只报告几何事实，不代表命中率、武器射程、穿透、骰子、伤害或最终“能否射中某个目标”。

#### 复杂掩体的几何与规则边界

- 《赛博朋克 RED》的掩体是二元语义：攻击者能看见目标时，目标就不在掩体中；不能用“半掩体百分比”解释红/绿线。
- 2026-07-18 用户选择临时保守方案：汽车、栏杆等复杂静态资产先用 `OcclusionGeometry` 从视觉模型子树计算物件根局部边界，再生成一个独立、可随物件旋转的盒形 CombatBody。该盒故意封住破窗、空驾驶舱和打开的车顶盖缝隙，让测试阶段不会因为视觉网格洞口反复穿线。
- CPR 的完整掩体要求整个人不可见。当前 Gvtt 采用更简单的桌面工具规则：静态物件真实视觉高度严格高于 `1.0` 米就视为可提供二元掩体；低于或等于 `1.0` 米就不作为战斗掩体。高度只用于分类，一旦分类为掩体，挡枪线按旋转后的地面粗框判断，不再因为射线高度高于车顶而放行。这是 P2.4 的临时 2.5D 抽象，不代表模型真的变高，也不包含蹲伏、爬高、穿透或命中率规则。
- 这个粗框不是 `PickProxy` 的 AABB，也不读取或复用拾取代理的数据；两者只是可能都呈盒状。车门挡、玻璃不挡、引擎盖/车顶盖分区这类精细部件与材质识别不进入 P2.4，也不进入 P2.5 的迷雾基础版。未来只有在 GM 明确需要时，才另开“手工标注低多边形战斗部件”任务；项目不做自动理解模型材质、机械结构、防弹区域或穿透规则。

#### 验证方法

- 自动回归必须覆盖：第 20/21 层隔离、射手排除、普通 Token 不挡枪、汽车真实高度保持约 `1.47` 米、`0.99` 米和 `1.00` 米不挡枪、`1.01` 米开始挡枪、高于汽车的测试线仍被地面粗框挡住、旋转盒不出现世界 AABB 假阳性、瞄准独占选择、右键环绕可共存、退出后恢复选择，以及锁定前后终点保持。
- 可见窗口验收：选中一个 Token 后移动鼠标，应只出现一条线和一个青色上半圆；方向扫过汽车时应出现橙色命中段与灰色余段。瞄准时直接左键汽车，当前选中仍应是 Token 且战斗线锁定；第一次 `Esc` 解锁、第二次退出瞄准，再点汽车才应切换选择。
- P2.5 只复用 `OcclusionGeometry` 的源几何生命周期，不复用 `CombatLinePreview`、`CombatBody` 或单条三维射线算法；迷雾必须用自己的地面轮廓和可见区域结果验证。P2.5 不补做汽车玻璃/车门/空舱等精细材质判断，避免把战争迷雾误扩成资产语义编辑器。

### P2.5 LOS 基础闭环

#### 当前范围

- 运行态当前选中或正在移动的 Token 自动成为唯一玩家视角来源，不要求 GM 点击“设为视点”。所有从 Token 栏放置的对象都挂 `TokenProperties`；编辑态点击/Gizmo 移动只用于备团编辑，不接管玩家视点、不重算运行态迷雾。启动运行态时尚未选择则暂取场景树第一个 Token，运行态选择非 Token 或空地不清除当前视点，当前 Token 删除后顺延，没有 Token 时关闭迷雾计算。多 Token 视野合并留到 P7。
- 只有墙体生成 `LOSOccluder`。交互物体、地形、装饰、汽车、Token 和光源不自动进入 P2.5，避免把 P2.4 的临时战斗粗框错误继承成视线语义。
- 基础版没有规则视距，视野以当前地图矩形为外边界。`WallProperties.blocks_los=false` 的墙不提供遮挡线段。
- `LOSOccluder` 直接调用 `OcclusionGeometry.get_local_bounds()` 和 `get_local_ground_footprint()`，把局部四角按墙根的世界变换转换成 XZ 地面线段；不创建物理体，不查询 PickProxy 或 CombatBody。
- `LOSService` 是 Main 场景下的运行时服务，不是 Autoload。它保存主摄像机引用、注册墙体遮挡体并缓存最终世界可见多边形；墙增删/变换/开关、地图尺寸或摄像机中心地面点变化时才重算，静止帧不重算几何。

#### 可见区域与投屏显示

- 可见多边形采用活动边最小堆旋转扫描。墙段先在交点处分割并去重，再加入地图四边；扫描只在最近活动线段变化时输出顶点，避免简单端点射线在深凹角把两个交点跨墙直连而漏视。
- 主窗口和投屏 `Window` 各挂一份显示遮罩，共用同一份 `LOSService` 世界多边形。透明 `SubViewport` 用 `Polygon2D` 画白色可见区，全屏 `TextureRect` 材质把白区外压暗。主窗口固定按“玩家暗区层 0 → GM 工具前景层 1 → GM 操作 UI 层 2”合成：地图按玩家规则变暗，Gizmo（变换手柄）、选择框、按钮和属性面板保持清晰。
- `GMToolOverlay` 使用共享主窗口 `World3D` 的透明 `SubViewport`，镜像主摄像机并只渲染 GM 专用第 20 层，再把结果放到暗区之上；Gizmo 的旋转辅助线 `Control` 同样归入该前景层。它只存在于主窗口，不接入 `CastView`，投屏继续排除第 20 层和所有 GM 工具。
- 观察 Token 的世界 XZ 坐标变化时才重算世界可见多边形；静止帧不重算。摄像机移动不改变观察点，只让主窗口和投屏各用所属 `Camera3D.unproject_position()` 重投影同一结果。
- P2.5 的地面投影遮罩不读取三维深度。倾斜摄像机在可见边界附近可能让高模型少量越界露出；深度感知合成明确留到 P7。

#### P2.6 墙体破坏联动点

- `WallProperties.set_blocks_los()` 是编辑墙体基础 LOS 语义的统一入口并发出 `los_blocking_changed`；`LOSOccluder` 订阅后立即清空或恢复线段，`LOSService` 随后重算。
- P2.6 以 `wall_state` 和基础 `blocks_los` 推导运行态有效值：`wall_state != DESTROYED && blocks_los`。破坏时只把有效值传给 `LOSOccluder.set_blocks_los(false)`，不得覆盖基础字段；修复时重新按基础值同步。挡枪层采用同样的独立推导，玻璃墙等特殊墙体才不会被错误恢复成挡视线。
- P2.6 不应删除 `LOSOccluder` 组件或让破坏控制器直接操作可见多边形；只改变墙体状态和有效开关。
- 移动层同样从 `wall_state` 推导有效阻挡：`MovementService` 收集运行时几何时跳过 `DESTROYED` 墙体，既不生成 `MovementObstacle`，也不生成 `MovementBody`。破坏/修复成功后由 `main.gd` 重建移动服务，清空旧体型缓存和活动路径；读档后进入运行态的既有重建会自然应用保存的墙体状态。

#### P2.6 最小闭环（2026-07-18 已落地）

- `WallProperties.wall_state` 是权威状态。`WallStateController` 只接受带 `WallProperties` 的墙体根节点；破坏要求 `destructible=true`，重复破坏返回无变化，修复只接受已破坏墙体。
- 破坏写入 `DESTROYED` 和 `durability_current=0`，隐藏完整墙体根，关闭有效 `CombatBody` 与 `LOSOccluder`；修复写回 `INTACT`、把当前耐久恢复到最大值，并按基础 `blocks_shot` / `blocks_los` 恢复阻挡。墙体根和专属组件始终保留，不使用 `queue_free()`。
- 编辑态必须显示已破坏墙的完整模型，供 GM 选中、移动和修改属性；这只是编辑视图覆盖，不把 `wall_state`、耐久、LOS、枪线或移动阻挡改回完整。再次进入运行态时按 `wall_state` 重新隐藏模型。`WallStateController.sync_wall(root, show_destroyed_visual)` 统一执行该表现，模式切换、编辑属性同步和读档迁移都必须传入当前模式。
- 持久字段为 `wall_state`、`destructible`、`durability_max`、`durability_current`、`blocks_los`、`blocks_shot`、`cover_level`。`visible`、战斗碰撞层、LOS 线段和移动导航几何都是派生运行状态；读档补齐组件后统一调用 `sync_wall()`，进入运行态时再重建移动服务，不把运行时 `StaticBody3D` 或导航 RID 存入场景。
- 最小视觉只隐藏完整墙体模型。Mesh 碎裂、预碎块替换、刚体碎块、粒子、音效、复杂光照/GI 重算全部后置；破坏插件不得拥有状态或删除墙体根。
- 墙体状态切换使用 P2.2 既有全量重建链。静态场景解析仍在主线程执行；当前开发场景运行态实测破坏重建约 2.02 秒、修复重建约 1.81 秒，属于明确可感知停顿。基础版先保证拓扑正确；后续性能阶段升级为分块导航网格或异步执行可异步的烘焙部分，不以动态避障替代拓扑更新。

#### 验证方法
