## 一、项目本质

**不是另一个 Foundry 替代品，而是一个"3D 交互地图引擎"。**

核心定位：

- 俯视 3D 地形（像看实体桌面）
- 实时光照 + 视线遮挡
- 可交互地图（墙可破坏、地形可改变）
- 2D 立绘 Token 站在 3D 地形上
- 双模式：线下投屏（实物棋子）+ 网团（电子 Token）

**锦上添花，不是推翻重来。** 传统跑团已经很好了，电子化是要去掉一些计算摩擦、增加视觉沉浸体验，不是要取代真人 GM。

---

## 二、产品哲学

### 用户是谁

| 角色 | 要什么 |
|------|--------|
| **GM（主持人）** | 不打断节奏、直觉操作、减少计算负担。但不是取代他的判定权 |
| **玩家** | 沉浸感（好看）、自由度（不被软件限制） |
| **开发者（你）** | 做一个自己用着爽的开源东西 |

### 原则

- 开源（MIT），很长一段时间不卖钱
- 核心卖点是地图交互 + 光照 + 特效，不是规则自动化
- 引擎是建议，不是取代——最终判定权永远在 GM 手里
- 先做给自己用，做爽了再考虑别人

## 三、用户流程

```
打开软件
    ↓
选择规则系统（内置通用模板 / 自定义规则包 / 无预设手动模式）
    ↓
搭建/导入 3D 俯视地图
    ↓
放置 Token（电子模式 / 实物模式）
    ↓
设置环境光照（暗夜 / 地牢 / 日光 / 手动布光）
    ↓
开始跑团
    ├── 移动 Token
    ├── 操作（攻击 / 施法 / 技能）
    ├── 检定（软件自动掷骰 / 手动输入实物骰结果）
    ├── 特效（火球爆炸 / 光照瞬间变化）
    ├── 地形变化（墙被破坏 / 火把点燃）
    └── GM 手动介入（规则未覆盖的边界情况）
```

## 四、规则系统架构

### 三层模型

```
┌─────────────────────────────────────┐
│      ① 引擎通用能力（你写）           │
│  地图 / 光照 / Token / 骰子 / 特效   │
│  状态管理 / 事件系统 / 计时器        │
│  — 不绑定任何具体规则系统              │
├─────────────────────────────────────┤
│      ② 规则配置（JSON 文件）          │
│  D&D 5e / COC / Fate / 自创         │
│  独立的配置文件包，引擎读取后行为      │
│  — AI 辅助生成，人工审核               │
├─────────────────────────────────────┤
│      ③ GM 现场判断                   │
│  规则未覆盖的边界情况 → 弹窗/手动输入  │
│  — 刻意保留的空间，不是"没做好"        │
└─────────────────────────────────────┘
```

### "表"的讨论（2026-07-01）

| 观点 | 来源 |
|------|------|
| 规则可以用一张通用表来描述 | 用户提出 |
| AI 读规则书自动填表，软件直接跑 | 用户提出 |
| 反驳：八种不同检定模型难以统一描述 | Claude 反驳 |
| 反驳：AI 填表准确率不到开团级别 | Claude 反驳 |
| 反驳：通用表到复杂逻辑退化成了写代码 | Claude 反驳 |
| 反驳：抽象表的诱惑会推迟做地图核心功能 | Claude 反驳 |
| 结论：用三层模型替代通用表 | 共同结论 |

## 五、规则框架

### 顶层分类

```
TTRPG 规则总框架
│
├── 主规则（引擎逻辑 + 配置）
│   ├── 核心检定
│   │   （用什么骰子、公式怎么算、如何判成败）
│   ├── 战斗规则
│   │   （先攻、回合结构、动作经济、死亡）
│   ├── 魔法规则
│   │   （施法机制、法术位/MP、专注）
│   ├── 环境规则
│   │   （光照、遮蔽、困难地形、距离）
│   └── 叙事规则
│       （经验、社交、休整）
│
└── 角色规则（数据）
    ├── 属性（基础能力值）
    ├── 资源（消耗性数值：HP、MP、法术位……）
    ├── 技能/熟练（角色会什么）
    ├── 专长/特性（特殊能力）
    ├── 装备（武器、护甲、魔法物品、消耗品）
    └── 成长（经验值、等级、升级收益）
```

### 特殊机制（横跨多个模块的综合调用）

某些机制不单纯属于某一类，而是调用多个组件的流程：

```
变形（Wild Shape）= 属性替换 + 资源替换 + 特性切换 + 操作限制 + 计时
专注（Concentration）= 状态管理 + 施法限制 + 受击中断
施法（Spellcasting）= 资源消耗 + 目标选择 + 范围/伤害 + 特效播放 + 状态变更
```

## 六、D&D 5e 示例拆分

### D&D 5e 在"主规则"里有什么

| 规则块 | 内容 |
|--------|------|
| 核心检定 | d20 + 加值 vs DC、优势/劣势、自然1/20 |
| 战斗规则 | 先攻 = d20 + DEX、动作+附赠+移动、攻击 vs AC、伤害类型、死亡豁免 |
| 施法规则 | 法术位系统、专注、法术成分(V/S/M)、范围(锥体/球体/线/自身)、施法时间 |
| 冒险规则 | 休息(短休/长休)、光照等级(明亮/微光/黑暗)、掩蔽等级 |

### D&D 5e 在"角色规则"里有什么

| 数据块 | 内容 |
|--------|------|
| 属性 | 力量/敏捷/体质/智力/感知/魅力 (3-18)，加值 = (属性-10)/2 |
| 资源 | HP、AC、法术位(每环若干)、生命骰、熟练加值、每日特性次数 |
| 技能/熟练 | 18项技能 + 工具 + 武器护甲熟练 |
| 装备 | 护甲(提供AC)、武器(伤害骰+属性)、魔法物品、消耗品 |
| 成长 | 经验值与等级、专长/属性提升、职业特性等级表 |

## 七、核心卖点验证

### 实时光照效果

- 引擎原生 3D 光照管线，解决 2D VTT 没有真实光照的问题
- 火炬照亮黑暗、法术爆发瞬间照亮、墙后暗区——目前 VTT 无原生实现

### 地图交互性

- 墙可以被破坏 → 光照从缺口穿过
- 可用插件：Godot-Destruction（2026.1 更新，支持 Voronoi 碎裂 + Stencil 穿孔）
- 不是预设动画片段，是运行时实时计算

### 法术特效

- Godot 生态已有成熟粒子系统 + 商业 VFX 包（Binbun/VFEZ-godot）
- 火球/闪电/治疗等效果拖入即用，不需要自己从头搓粒子

### 技术选型

- **引擎：Godot 4.4+**（MIT 协议、Vulkan/Metal 原生渲染）
- **破坏系统：Godot-Destruction**（开源，实时 Mesh 碎裂 + 穿孔）
- **特效系统：VFEZ-godot + GPUParticles3D** + 商业 VFX 包
- **3D 资产管线：AI 生成（Meshy/Rodin/Luma）+ 人工审核**

## 八、已排定不做的事（初期）

- ❌ 不替代 GM 判定（GM 永远是最终决定者）
- ❌ 不做规则编辑器（通过配置 + AI 生成替代）
- ❌ 不做联网同步（先做好本地单机）
- ❌ 不做模组市场
- ❌ 不与 Foundry 正面竞争规则自动化深度
