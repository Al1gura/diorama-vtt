# Gvtt 项目指令

## 项目定位
- 不是游戏，是用 Godot 做的 GM 桌面工具——开源 3D 俯视交互地图引擎
- 首要场景：线下跑团，GM 电脑操作，投屏给所有人看
- Windows 单目录便携应用，下载一个 ZIP、解压即用，无需安装；核心资源内嵌，原生扩展 DLL 与 EXE 同目录。GM 一人操作。不做：联网同步、骰子/规则系统、模组市场、玩家端、账号

## 命名规范
| 元素 | 格式 | 例 |
|------|------|----|
| 文件名/目录 | `snake_case` | `player_controller.gd` |
| 场景节点名 | `PascalCase` | `PlayerSprite` |
| 脚本变量 / @export | `snake_case` | `grid_size` / `@export var max_health: int` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_HEALTH` |
| 信号名 | 过去式 `snake_case` | `signal health_changed` |
| 私有成员 | `_` 前缀 | `var _current_health` |
| 所有变量 | 必须静态类型注解，不准 `:=` | |

## 协作方式

### 沟通
- 立即回应用户，不闷头操作。任何连续工具调用之间必须输出进展说明。**禁止沉默操作。**
- 用户明确交代功能、修复或文档目标后，直接连续完成调研、修改、验证和日志维护，普通文件修改不重复索要确认。调研回执、源码对照卡、计划说明都是进展汇报，不是暂停等待点；汇报后必须继续执行。只有目标或范围无法可靠判断、操作可能破坏数据、需要额外系统权限时才停下来询问。用户只说"喂"或"？"时先回应，不擅自开始新任务。
- 用户单独回复 `1`，表示“好了／同意，继续执行”。直接继续当前已说明的下一步，不再追问确认。
- 用户无编程基础。专业问题用通俗语言，能打比方就打比方。**英文术语必须加括号写中文翻译**，不准单独出现英文。

### 任务报告
- 每次任务完成后立刻报告：做了什么、没做什么及原因、产出物在哪。不啰嗦。完成时刻就说，不让用户追问。
- 每次做好功能或修好 bug 后，必须明确告诉用户如何验证工作成果：自动测试跑了哪些、用户在 Godot 可见窗口里应该点哪里/拖哪里/按什么键、预期应看到什么、若仍需人工体感确认也要直说。
- **先说出要做什么，再做。** 不要默默在后台动手。

### 开发日志维护
- 每次会话结束前更新 `devlog/DEVLOG.md`：已完成任务、进度变化、新问题及状态。
- 格式 `- [x]` / `- [ ]`，问题追加到表格（标注日期和状态）。

### 辩证思考
- **不顺着说。** 用户观点先找漏洞、矛盾、其他可能性。
- **看本质，不只盯字面。** 用户说"去掉A"，不一定是A错了——可能是分类方式不同、或测试你的理解。
- **疑问句不等于"我不知道"。** 可能是在测试你、或引导你发现矛盾。先理解意图再回应。
- 指出用户前后说的冲突、或需求与已有结论的矛盾。

## 信息核查
- 回答项目状态/配置/进度时，交叉比对以下文件；矛盾处明确指出不能隐瞒：
  `docs/design.md` · `docs/CCxGodot.md` · `devlog/DEVLOG.md` · `.agents/skills/` · `.codex/` · `addons/` · `project.godot` · `scenes/main.tscn` · `scripts/main.gd`
- **每个论点必须有依据——搜索到的事实或基础原理推导，不能凭感觉。** 回答问题必须引用来源。
- 不能单凭记忆回答，必须有文件读取或搜索结果的产出物作证。
- **Godot API 必须有离线文档依据。** 写代码前搜索 `reference/Godot 4.7 Dooc/` 确认类名、方法签名、属性名，不准编造。
- **Godot 3→4 API 改名高危。** 给 Control/Button 等老类赋属性时（`hint_tooltip`、`flags_unshaded` 等 3.x 旧名），先查离线文档确认 4.7 的现行名称与类型。赋值非法会**静默中断**当前函数（不一定打红字 parse error），后面代码全不执行。
- **"跑游戏无红字"不等于"代码跑完整"。** 静默中断不打 parse error。判函数跑没跑到底，必须靠 `game_eval` 看运行态结果。

## 现成方案先行门禁（最高优先级）
- 本门禁同时适用于：回答方案问题、功能设计、bug 修复、性能/手感优化、编写或修改代码。“这是否常见”“网上有没有现成办法”“为什么这样设计”也属于方案决策，不得因为暂时不写代码而跳过调研。
- 用户提出任何功能、交互、Godot 行为、架构或常见做法问题后，第一条进展必须明确说明：本轮是否触发现成方案调研、将核查哪些来源、调研完成前是禁止下结论还是禁止写代码。
- 给出方案结论或修改项目文件前，必须完成并向用户公开“调研回执”：

| 层级 | 必查来源 | 回执必须包含 |
|------|----------|----------------|
| 项目现状 | 当前代码、场景、文档 | 现有实现和真正问题 |
| Godot 源码 | 项目当前精确版本源码 | 标签/提交、文件、类、函数、调用顺序 |
| 官方资料 | Godot 官方文档和演示项目 | 官方支持的做法和适用范围 |
| 社区方案 | 英文社区、插件、开源项目 | 可复用程度、Godot 版本、维护状态和冲突风险 |

- 只搜索但不向用户公开来源和实际查得内容，视为未完成调研。只找到单个 API、单篇文档或单个社区回答，也视为未完成。
- 调研回执未完成时：不准给出确定性方案结论，不准修改相关功能代码；必须明确报告缺少哪一层并继续搜索。调研回执完成后，如果用户已经明确要求开始该功能/修复，禁止再次询问“是否继续”，应直接进入实现、验证和日志维护。
- 可免除本门禁的情况只有：纯文字润色；仅从已有文件直接查证项目状态；用户明确要求不搜索；与技术方案无关的普通交流。
- 调研时先查现成做法，不把“只是 bug / 小补丁 / 体验优化”当作跳过理由；先看 Godot 自己怎么做、官方如何规定、社区成熟模式怎么做，再决定采用插件、复用社区模式或自己实现。不准在验证方法不明确时动手。
- **涉 Godot/GD 既有行为时，Godot 源码是硬门槛。** 只要用户说“按 Godot/GD 的方式”“看看 GD 怎么做”“照 Godot 编辑器”“抄现有行为”，或功能明显在复刻 Godot 编辑器行为（拖放、选中、gizmo、导入、资源面板、3D 视窗交互等），必须先拿到 Godot 引擎源码中的具体文件、类/函数名和完整行为链，再设计实现。拿不到时必须报告“源码依据未完成”并继续查，不得用文档、API 推断、用户实测或社区答案替代。
- **不准把“行为相似”说成“参考了 Godot”。** 只有读过 Godot 源码并能指出来源文件/函数时，才允许这样表述；否则必须明说“这是近似方案，尚未完成 Godot 源码对照”。
- **调研依据要写进交付。** 功能任务的完成报告和 `devlog/DEVLOG.md` 必须记录四层来源、采用了哪些思路、哪些没采用及原因；缺少记录不能判定功能完成。

### Godot 源码一致性门禁（实现前第二道门）
- 涉及拖放、选中、导入、资源缓存、资源面板、Gizmo、3D 视口、场景实例化等行为时，自动进入“Godot 源码一致性任务”；不需要用户再次提醒“按 Godot 做”。
- 必须锁定项目当前 Godot 精确版本对应的源码标签或提交。本项目默认使用 `4.7-stable`；禁止用 `master`、其他版本或仅官方文档代替。确需参考其他版本时必须单独标明差异，不能混作 4.7 依据。
- 执行任何项目文件写入前，必须先向用户提交“源码对照卡”，包含：
  1. 要复刻的完整行为链，而不是单个 API 或函数。
  2. Godot 源码标签或提交号。
  3. 具体源码文件、类、函数和调用顺序。
  4. Godot 在哪个阶段完成加载、转换、缓存、实例化和清理。
  5. 本项目每一步对应的文件和函数。
  6. 无法照搬的差异、原因和替代方案。
  7. 每项行为对应的自动测试与运行态验证方法。
- 缺少以上任意一项，只允许继续查资料，禁止修改相关功能代码。用户已经明确授权当前功能或修复目标时，源码对照卡提交后必须直接继续实现，不再要求用户回复固定口令；不得把“提交源码对照卡”当成等待确认。只有对照结果暴露出新的范围选择、数据风险、系统权限需求或与原需求冲突时才暂停询问。
- 源码核查必须覆盖完整调用链。只找到某个 API、某个相似函数或单个局部实现，不得据此宣称“按 Godot 做”。
- “使用了相同 API”“效果看起来相似”“无红字”“测试通过”都不能单独证明与 Godot 行为一致。
- 完成报告必须提供“Godot 源码行为 → 本地实现 → 自动测试/运行验证”的逐项映射。缺少映射不得宣布完成。
- 实现后才发现遗漏的 Godot 行为时，必须撤销此前的“已完成”结论，在 `devlog/DEVLOG.md` 记录流程违规和补救结果，不能包装成普通后续优化。
- 搜索关键词**必须覆盖英文社区**，中文资源远少。
- 找到现成方案后验证兼容性（4.x 而非 3.x、维护状态、与现有插件冲突）。
- **优先用插件 → 社区已验证模式 → 最后自己写。** 自己写的先在编辑器测无报错，再请用户实操确认（我无法截图，最终以体感为准）。
- **用户说"去看/搜/查/抄某来源"=停止符和转向指令。** 停下手头所有事，按用户指定方向查证。
- **不准逃避问题。** 用户说找不到 AssetLib 标签页，就帮用户找到它，不能改用手动下载 ZIP 绕过去。

## 调试纪律
- **运行态探查不通先解通道，别在代码上绕。** 调运行态工具前先确认通道通：跑游戏后看 `helper_live`、`logs_read(source=game, count=80)` 有没有行。不通就先看 Godot 启动报错有没有 `port 6006/6005 Already in use`。有=僵尸进程占端口，排障：①任务管理器结束多余 Godot 进程 ②重启电脑。MCP 断连优先重启 Cowork 客户端而非 Godot（见 memory `gvtt_mcp_reconnect_shortcut`）。通道不通加诊断是白费。
- **拿到 Godot 报错文本前不改代码、不加诊断。** 没有实测依据的诊断都是赌。
- **交接总结的假设当"待验证"，不当结论继承。** 第一轮先逐条质疑、用实测确认。
- **`game_eval` 的 code 参数缩进必须用 tab（`\t`），不准用空格。** 包装器每行前加一个 tab，内层空格→混合缩进→`EVAL_COMPILE_ERROR`。
- **用户问"为什么你要做这件事"=自查是否绕远的触发器，不是要你解释。** 停下来想"是不是把简单搞复杂了"。
- **重启电脑是特定症状的解法，不是万能兑底。** 只在「运行态探查全废 + port 6006/6005 Already in use」这个组合下推荐。代码/UI bug 重启没用，该查的代码必须查。

## Godot AI 工具规范

### MCP 按需使用边界
- **MCP 是 Godot 运行态开发与验收通道，不是所有任务的常驻前置。** 连接本身会增加工具调用、日志/场景树上下文、等待时间、端口与编辑器状态维护成本，只在这些成本能换来运行态证据时使用。
- 文档整理、JSON/清单分析、普通源码搜索、静态检查、纯文件操作和用户明确要求的数据清理，优先使用文件工具与命令行；这类任务不为“形式上经过 Godot”而启动编辑器或连接 MCP。数据清理时若 Godot 可能写回目标目录，必须先关闭 Godot。
- 涉及 `Control` 界面、场景树、拖放、输入、Gizmo、3D 视口、投屏窗口、音视频、节点显隐、信号时序或其他运行态行为时，必须优先连接 Godot MCP，并使用 `project_run`、`helper_live`、`logs_read`、`game_eval`、`script_manage` 等真实入口取得证据。
- 每个相关任务开始时先检查当前对话是否实际暴露 Godot MCP 工具。项目安装了 `addons/godot_ai` 不等于当前 Codex/Cowork 对话已经连接 MCP；不得把插件存在、Godot 进程存在或命令行测试通过冒充 MCP 已连接。
- MCP 不可用时可以继续做不依赖运行态的文档、文件实现和自动测试，但必须立即向用户说明缺少哪些工具；涉及可见界面或运行态的任务不得宣布完整完成，必须保留“恢复 MCP 后 `game_eval`/可见验收”的待办。
- Godot 命令行无窗口/Windows 自动回归是稳定的补充证据，不能替代 MCP 运行态探查或 GM 真人点击；反过来，MCP 单次探查也不能替代可重复自动测试。
- 完成一次 Godot 功能的推荐顺序：文件修改 -> 自动回归 -> 按需连接 MCP -> `game_eval`/场景树/日志实证 -> Windows 可见窗口 -> GM 真人体感确认。任务结束后不为无关工作维持常驻连接。

### 批量操作
- **场景修改合并为一次 `batch_execute` 调用。** 删除/创建节点、设属性、保存场景都放一个 `commands` 数组。
- `batch_execute` 用**插件命令名**（`create_node`、`delete_node`、`set_property`、`save_scene`），不是 MCP 工具名（不是 `node_create`、`scene_save`）。
- `save_scene` 做 `batch_execute` 最后一条，不在之后单独调。
- 原因：当前不支持 MCP 工具"总是允许"（GitHub issue #29026, #25966），batch_execute 是此限制下的最优解。

### 禁止断链 MCP 连接
- **不准调用会断开 MCP 连接的工具**（`editor_reload_plugin`——官方写明 `transport_will_drop: true` 等）。
- **遇到 GDScript 脏缓存报错**（行号对不上磁盘代码、File System 重扫不清）时，**不准用断链手段冲**。正确顺序：
  1. `game_eval` 实证检查脚本是否正常运行（helper live=实际加载的是好的，编辑器报错是缓存脏数据）
  2. 读磁盘脚本核对报错行号：不对应→缓存脏数据，承认"编辑器报错、实跑正常"
  3. 确需清缓存→先明确告知风险并征得同意，否则不准做
- 只在温和办法全试过 + 用户明确同意后才允许断链。
